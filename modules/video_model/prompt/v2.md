You are implementing **Module B V2** of the Live-Document project.

The current Module B already provides an end-to-end explanation-video pipeline using Wan2.1-T2V-1.3B, a procedural fallback, ffmpeg post-processing, ffprobe validation, metadata logging, and MP4/WebM/GIF export.

Do not rewrite the project from scratch. First inspect the existing implementation, tests, README, output structure, and shared interfaces. Preserve all useful V1 functionality.

# 1. V2 objective

Add and optimize **LTX-Video 0.9.8 2B distilled** as the primary probabilistic video backend for explanation-oriented short videos on:

* AMD Radeon Pro W7900
* 48GB VRAM
* Linux
* ROCm-compatible PyTorch
* Python 3.10+

The V2 goal is not to add many product features.

The goal is to establish a measurable optimization loop for:

1. faster generation;
2. better explanation-oriented video quality;
3. reproducible comparison between configurations;
4. clear evidence for the best speed-quality trade-off.

The system should continue producing:

* 3–8 second MP4;
* WebM;
* GIF;
* complete metadata JSON;
* failure records.

# 2. Product context

The upstream document module produces an `ExplanationSpec` or equivalent JSON containing fields such as:

```json
{
  "source_text": "Water evaporates, rises, cools, and condenses into clouds.",
  "visual_goal": "Show the transformation and upward movement of water during evaporation and condensation.",
  "content_type": "process",
  "video_prompt": "A clean educational animation showing water evaporating upward and condensing into a cloud."
}
```

Module B should generate short, clear, explanation-oriented visual motion.

It is not a general cinematic video generator.

Prioritize:

* stable objects;
* fixed or nearly fixed camera;
* one clear visual relationship;
* minimal clutter;
* simple motion;
* no generated text;
* easy conversion into a short looping visual.

# 5. LTX backend implementation

Implement an LTX backend using the most practical officially supported Python or Diffusers-compatible path available in the environment.

Requirements:

* support ROCm-compatible PyTorch;
* do not assume CUDA-only kernels;
* avoid mandatory xFormers, FlashAttention, NVIDIA FP8, or CUDA extensions;
* use FP16 or another W7900-compatible precision;
* record `torch.version.hip`;
* record the detected GPU name;
* fail clearly when ROCm or model dependencies are unavailable.

The LTX implementation should support, where feasible:

* text-to-video;
* image-to-video as an optional or experimental path;
* deterministic seed;
* configurable frame count;
* configurable inference steps;
* configurable resolution;
* configurable guidance;
* model reuse across multiple generations;
* optional VAE tiling or slicing;
* optional CPU offload, but disabled by default on the 48GB GPU.

Do not introduce unsupported optimizations merely because they work on NVIDIA.
