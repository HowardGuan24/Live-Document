
# Codex 第一版 Prompt（概率生成：视频模型到 GIF）

````markdown
You are helping build Module B of our MVP: **probabilistic generation for document-to-GIF explanation**.

## Project context

We are building a system that takes a document, identifies a passage that is worth turning into a dynamic explanation, and then generates a short visual explanation.

My module is the **video-model-based generation path**.

This module receives structured output from the document module and should generate:
1. a short video (3–8 seconds),
2. a GIF or WebM,
3. metadata including generation parameters, runtime, and failure notes.

We are **not** building a general creative video generator.  
We are building an **explanation-oriented short video generation pipeline**.

The focus is:
- short, clear, explanation-friendly motion,
- prompt optimization for explanation,
- stable generation workflow,
- export to GIF/WebM,
- recording failures and limitations.

## Environment

- Linux environment
- One AMD GPU with 48GB VRAM
- You may choose open-source models and tools that are realistically runnable in this environment
- Prefer solutions compatible with **AMD / ROCm**
- If necessary, compare 2–3 candidate models/workflows and pick one practical first version
- Do not assume NVIDIA/CUDA-only tools unless you clearly explain why and provide a fallback

## Input format

Assume the document module outputs JSON like this:

```json
{
  "source_text": "Gradient descent updates parameters along the negative gradient direction.",
  "visual_goal": "Help the user understand that a point moves step by step downhill on a loss curve.",
  "content_type": "process",
  "video_prompt": "A clean educational animation showing a point moving step by step down a smooth loss curve toward a minimum, with simple labels and no unnecessary background clutter."
}
````

At minimum, your pipeline should use:

* `source_text`
* `visual_goal`
* `video_prompt`

## Output requirements

For each run, produce:

1. a short video file (preferably mp4 or webm),
2. a GIF or WebM suitable for embedding,
3. a metadata JSON file containing:

   * input JSON,
   * chosen model/workflow,
   * generation parameters,
   * runtime,
   * output file paths,
   * success/failure status,
   * notes about visual quality or explanation issues.

## Main tasks

Please implement a **first runnable version** of this module.

### Task 1: model/workflow selection

Choose a practical open-source workflow for short explanation video generation on AMD GPU.

Possible directions include:

* text-to-video,
* image-to-video,
* text-to-image + image-to-video,
* ComfyUI-based workflow,
* diffusers-based workflow,
* or another practical approach.

Your job is to decide what is most feasible for a first MVP.

Important:

* Prefer stability and feasibility over ambition.
* If pure text-to-video is too unstable, it is acceptable to use a two-stage pipeline:

  1. generate a key image or frame,
  2. animate it into a short video.

### Task 2: explanation-oriented prompting

Design a prompt strategy specifically for **explanation videos**, not artistic videos.

The prompt strategy should try to encourage:

* clear subject,
* minimal clutter,
* simple motion,
* stable objects,
* fixed or nearly fixed camera,
* short loop-friendly actions,
* educational / diagram-like style when appropriate.

Also document common prompt patterns for different content types, e.g.:

* process,
* state change,
* data flow,
* scene / analogy.

### Task 3: generation pipeline

Build a script or small module that:

* reads the input JSON,
* prepares the prompt(s),
* runs the selected generation workflow,
* saves the video,
* converts or compresses to GIF/WebM,
* records metadata.

### Task 4: post-processing

Implement post-processing for explanation use:

* limit duration to around 3–8 seconds,
* prefer stable framing,
* crop or resize if needed,
* compress for easy embedding,
* if possible, improve loop quality.

### Task 5: failure logging

Since instability is expected, explicitly record failure cases such as:

* object drift,
* incoherent motion,
* bad loop,
* visually pretty but semantically unclear output,
* prompt ignored,
* output too noisy or cluttered.

## Deliverables

Please create:

1. `README.md`

   * explain chosen approach,
   * installation,
   * how to run,
   * model/workflow choice and why,
   * limitations.

2. main runnable script, e.g.

   * `generate_explainer_video.py`
     or a small package structure.

3. sample input JSON files

4. an output directory structure, e.g.

   * `outputs/video/`
   * `outputs/gif/`
   * `outputs/meta/`

5. a short experiment note or report, e.g. `notes.md`
   including:

   * candidate models considered,
   * chosen workflow,
   * what works well,
   * what fails,
   * which content types seem suitable for probabilistic generation.

## Suggested engineering constraints

* Keep the first version simple and runnable.
* Do not over-engineer a full product UI.
* CLI-first is fine.
* Prefer reproducibility.
* Save all parameters and prompts.
* Use Python where practical.
* Use ffmpeg for conversion if helpful.
* If ComfyUI is the best choice, automate it rather than relying only on manual GUI use.

## Success criteria for v1

A successful first version should:

1. run end-to-end from one input JSON,
2. generate at least one usable short explanation-style video,
3. export GIF or WebM,
4. log metadata and runtime,
5. document failure cases,
6. clearly explain which types of document content seem suitable for this probabilistic generation route.

## Important product perspective

This module is part of a larger system.
Our contribution is **not** “using a video model”.
Our contribution is to study and implement an **explanation-oriented video generation workflow** for document understanding.

So please optimize for:

* explanation clarity,
* short educational motion,
* workflow feasibility,
* stable outputs,
* clear reporting.

## Work style

Please:

1. inspect the environment,
2. choose the most feasible first version,
3. implement it,
4. explain tradeoffs,
5. do not silently assume unsupported dependencies,
6. keep the codebase clean and easy to hand off.

Start by proposing:

* the workflow you want to use,
* the candidate models/tools considered,
* and the exact v1 implementation plan.

Then implement the first version.

````

# 我对你这条线的一个额外建议

你是“视频模型生成 GIF”这条线，**第一版最好允许 Codex 选一个更稳的两阶段方案**，而不是强逼它纯 text-to-video。

也就是：

### 优先级建议

1. **文生图 / 图生图 先生成一个稳定关键帧**
2. **再做 image-to-video**
3. **最后转成 GIF / WebM**

因为这样更容易控制：

* 主体是谁
* 画面长什么样
* 背景别太乱
* 镜头别乱跑

这比纯 text-to-video 更适合你们“解释型视频”的目标。
